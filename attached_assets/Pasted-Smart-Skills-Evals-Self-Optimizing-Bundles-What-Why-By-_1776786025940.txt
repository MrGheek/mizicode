Smart Skills: Evals + Self-Optimizing Bundles

What & Why

By the time #38–#42 are in place, FLOATR will be able to:

import and rank skills
inject bundles into sessions
understand repo structure
shield context
govern memory and retrieval budgets

At that point, the missing piece is proof. FLOATR needs to measure which skills and bundles actually improve outcomes on a rented, hourly GPU session, then use that evidence to recommend better defaults over time. That matters especially in FLOATR because sessions run on paid GPU instances, can support multiple team members, and already track session lifecycle, spend, and memory across sessions.

This task adds:

evaluation runs
bundle scorecards
per-skill and per-bundle lift metrics
baseline vs bundle comparisons
a leaderboard
automatic recommendation weighting based on measured outcomes

The goal is to turn Smart Skills from “curated and plausible” into measured and self-improving.

Done looks like
Eval runs exist

FLOATR can execute a defined eval run against:

a single skill
a bundle
a baseline session with no Smart Skills
a bundle variant

Each eval run stores:

task type
repo context
token mode
model profile
runtime metrics
result metrics
qualitative outcome
Baseline vs treatment comparisons

FLOATR can compare:

baseline (no bundle)
default bundle
alternate bundle
bundle + extra skill
bundle - one skill

Comparison output includes:

time to first useful answer
total elapsed time
memory retrieved
context bytes/tokens injected
shielded output avoided
file/symbol search efficiency
success/failure
user rating if available
cost per useful outcome
Bundle leaderboard

A leaderboard exists for:

best overall bundles
best by task mode
best by repo shape
best by token mode
best by model family

Leaderboard is backed by actual stored eval data, not hardcoded opinion.

Skill contribution scoring

Each skill records:

activation count
eval appearances
positive lift count
negative lift count
confidence score
estimated contribution to outcome

This lets FLOATR demote weak skills and promote strong ones.

Recommendation weighting

Bundle compiler starts using measured performance as an input:

measured_lift
confidence
task-fit performance
repo-shape performance

This should influence but not fully override static compatibility/risk logic.

Eval jobs are async

Eval runs are asynchronous jobs, with statuses like:

queued
preparing
running
scoring
completed
error
Eval harness supports controlled variants

FLOATR supports simple experiment types:

A/B
ablation (remove one skill)
add-one
baseline-vs-bundle
Observability

FLOATR can answer:

which bundles are actually working
which skills cause regressions
whether Lean/Core/Ultra materially affect outcome quality
whether repo intelligence and memory governance improved throughput
Out of scope
Fully autonomous bundle rewrites with no review
Public marketplace rankings across all users
ML-based causal inference beyond simple contribution heuristics
Full visual experiment dashboard if not needed yet
Cross-tenant sharing of private eval results
Automatic mutation/generation of entirely new skills
Benchmarking every model/profile combination exhaustively
Steps
1. Add eval data model

Create tables such as:

skill_evals
bundle_evals
eval_runs
eval_run_variants
skill_leaderboard_cache or equivalent materialized view strategy

Store:

session/task context
bundle/skill composition
baseline linkage
metrics
score
confidence
notes
2. Define eval schemas

Standardize eval dimensions:

task mode
repo kind / size / language spread
token mode
model profile
team vs solo
bundle ID
variant type
3. Build eval runner

New service:

skills-evals.ts

Responsibilities:

schedule eval runs
run baseline and treatment variants
collect runtime metrics
persist scores
compute aggregate results
4. Add scoring model

For MVP, define a transparent composite score, for example:

success score
time score
retrieval efficiency score
context efficiency score
stability / error penalty
user feedback bonus

Keep weights explicit and configurable.

5. Add instrumentation hooks

Pull in metrics from the existing stack where possible:

session timing
memory usage / retrieval layer usage
context shield stats
repo retrieval hit stats
total session cost
bundle activation metadata
6. Add contribution heuristics

For MVP, use simple heuristics:

compare baseline vs bundle
compare bundle vs ablated bundle
attribute directional lift to removed/added skill cautiously
track confidence based on sample count

Do not attempt heavy causal modeling yet.

7. Add leaderboard computation

New service:

skills-leaderboard.ts

Compute:

top bundles by task mode
top bundles by repo size/shape
top individual skills by measured lift
skills with negative lift / regression risk
8. Feed bundle compiler

Extend skills-ranker.ts / skills-bundler.ts to use:

measured lift
confidence score
recency of results
negative lift penalties

Recommendation logic should remain safe:

low-confidence eval data cannot overpower trust/risk rules
experimental skills remain bounded
9. Add APIs + OpenAPI

Add endpoints such as:

POST /api/skills/evals/run
GET /api/skills/evals
GET /api/skills/evals/:id
GET /api/skills/leaderboard
GET /api/skill-bundles/leaderboard
GET /api/skills/:id/performance
GET /api/skill-bundles/:id/performance

Update OpenAPI and regenerate codegen.

10. Add first-party optimization skills

Add manifests for things like:

eval-aware-bundling
measured-recommendation-weighting

These are lightweight FLOATR-native skills representing the ranking logic, not user-facing gimmicks.

Design note

Do not make #43 into a giant benchmark lab. The point is to create enough measurement to:

improve default bundle selection
identify regressions
justify which skills deserve first-class status

This is the phase where FLOATR starts becoming self-improving instead of just feature-rich.
