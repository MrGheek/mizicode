These are good plans. The roadmap is coherent, and the last three tasks fit FLOATR’s actual shape well: shared GPU team sessions, persistent cross-session memory, and layered session orchestration.

My feedback is mostly about tightening boundaries and fixing a few dependency/reference issues before they go too far.

Overall

Three things jump out immediately:

Numbering/reference drift still needs cleanup

#42 says repo graph comes from #44
#42 mentions team/session model from #27
#44 mentions Memory Governance scope model from #60

Those should be normalized now, or Replit will wire things to the wrong assumptions.

Scope vocabulary should be unified across #42 and #44

#42 uses scopes like task → session → shared workspace → repo → user/operator → global
#44 uses session_core / repo_shared / lane_user / task

I would pick one canonical scope vocabulary now and use it in both tasks, otherwise you’ll end up migrating memory rows later just to rename concepts.

Task order
I would do:

#42 Memory Governance
#44 Team Intelligence
#43 Evals

Because #43 gets better once the lane model and shared-session coordination are real. Also, #44 does not really need to depend on #43.

#42 — Memory Governance + Budgeted Retrieval

This is strong. It fills a real gap and complements the earlier work well.

What I like
3-layer retrieval is exactly right
contradiction detection is valuable
TTL/decay + staleness + ROI is a strong combo
budget profiles tied to token modes is very on-brand for FLOATR
What I’d still tighten

Unify the scope model with #44 now
This is the biggest one. Don’t build one scope model for #42 and another for team lanes later. I’d strongly suggest you adopt the later lane-friendly vocabulary right now, for example:

task
lane_user
repo_shared
session_core
user_operator
global

or whatever naming you prefer, but pick one.

Make contradiction status explicit
Don’t just store a conflict_group_id. Also store something like:

conflict_status: open | reviewed | resolved | accepted_override

Otherwise contradictions become a pile of flags with no workflow.

Define what semantic contradiction detection does when embeddings are unavailable
Since you want local-first systems, specify:

lexical-only fallback is acceptable
semantic contradiction checks are optional/feature-flagged if embedding infrastructure is not ready

That avoids blocking saves on fancy logic.

Promoted items should still pass stale/conflict checks
If a note becomes a convention or a warning becomes a guardrail, it should not become “immune.”
Promoted items should still be:

stale-able
conflict-checkable
reversible

Be careful with automatic promotion
You already say threshold-based and reversible, which is good. I’d go one step further and say:

auto-promotion creates a candidate convention/guardrail
it does not become fully trusted until reviewed, unless confidence is very high

That keeps governance cleaner.

My verdict on #42

Very good, nearly ready.
The main thing is to unify the scope model with #44 before implementation.

#43 — Evals + Self-Optimizing Bundles

This is the right phase, but it needs the most operational discipline.

What I like
async eval runs
baseline vs treatment
leaderboards by task/repo/model
confidence-weighted ranking
low-confidence data not overriding trust/risk rules

All of that is sound.

What I’d still tighten

Add reproducibility fields
For every eval run, I would require:

repo commit SHA
skill version IDs
bundle version hash
model profile
token mode
prompt/runtime config version

Without this, leaderboard results become hard to trust.

Add GPU-hour guardrails
Since FLOATR runs on hourly rented GPUs, evals need budget discipline:

max concurrent eval jobs
per-run cost cap
daily eval budget
lower priority than live user work

Otherwise evals will quietly eat the value of the platform.

Make eval jobs contention-aware
If #44 is coming, evals should never starve live lanes.
Even before #44, I’d say:

eval jobs run at lower scheduling priority than interactive work

Be cautious about turning ranking logic into “skills”
This line:

eval-aware-bundling
measured-recommendation-weighting

is conceptually fine, but I’m not sure it should literally be exposed as user-facing skills. It may be cleaner as:

internal platform policies
compiler features

You can still model them as FLOATR-native skills if you want consistency, but I’d keep them hidden/internal.

Define success rubrics by task mode
A debug task and a review task should not share the exact same score weighting.
I’d define separate scoring presets for:

build
debug
review
refactor
team
My verdict on #43

Good plan, but add reproducibility + eval budget controls before build.

#44 — Team Intelligence + Multi-Lane Coordination

This is the most exciting one product-wise. It feels very aligned with how FLOATR is differentiated.

What I like
layered model is exactly right
lane overlays make sense
soft claims are the right choice
overlap detection via repo graph is strong
heavy-job scheduler is very important on a shared GPU
handoff signals are a smart minimal coordination layer
What I’d still tighten

This should not depend on #43
I would remove that dependency unless you explicitly want eval-backed lane policies before you build the lane system.
Team coordination can exist without leaderboards.

Your data model may be over-normalized for MVP
You probably do not need all of these on day one:

session_lanes
lane_assignments
lane_state
lane_claims
lane_handoffs
lane_heavy_jobs

For MVP, I would consider:

session_lanes
lane_claims
lane_handoffs
lane_heavy_jobs

and keep most state directly on session_lanes until real usage proves you need more tables.

Soft claims need expiry / heartbeat
Otherwise claims go stale fast in real team work.

I’d add:

claimed_at
last_heartbeat_at
expires_at
maybe claim_strength: watching | editing | owner

That will make conflict warnings much more useful.

Lane overlays must be lane-local, not globally injected
This is important.
If Alice is in a UX lane and Bob is in a debug lane, each should get their own overlay in their own prompt/runtime path. The whole session should not see all overlays all the time, or you’ll recreate context bloat.

Heavy-job scheduler should be fairness-based, not just priority-based
If you only prioritize debug/review-critical work, background indexing may never run.
I’d recommend a weighted fair queue:

priority
age
lane fairness
job class

Handoff notes should include structured type + free text
Not just free text. For example:

blocked
needs_review
safe_to_merge
watch_files
related_lane

plus optional notes.

My verdict on #44

Very good.
It just needs:

dependency cleanup
lighter MVP schema
claim expiry/heartbeat
explicit lane-local overlay behavior

For #43

For #44

These are strong plans. The architecture is now coherent enough that the main risk is no longer “wrong direction,” it’s schema drift and operational ambiguity.
