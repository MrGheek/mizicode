#111 — Kimi K2.6 Swarm Orchestration Layer

This is the most important one, and it is good. My main suggestions are about guardrails.

First, add a decomposition gate before spawning anything. Do not let every <decompose> block automatically become a swarm. Have the runner validate:

minimum task count, maybe 2–3
no duplicate subtasks
each subtask has enough specificity
total worker count stays inside a profile cap
the task is actually parallelizable

That protects you from over-eager decomposition and from models generating lots of tiny, noisy workers.

Second, make the decomposition format more structured than a raw XML-ish tag. I would strongly prefer:

one clearly delimited JSON block
required fields like id, task, goal, inputs, expected_output
optional priority
validation before spawn

That will make parsing, journaling, and synthesis far more reliable.

Third, worker prompts should be tightly scoped. Each worker should receive:

only the subtask
only the relevant shared context
a hard instruction that it is a worker, not the final synthesizer
no permission to spawn its own swarm

That will reduce drift and stop recursive behavior from sneaking in even if you keep recursive sub-swarms out of scope.

Fourth, add timeouts, retries, and partial-failure synthesis. Right now the plan says synthesize when all workers finish or fail, which is right, but it should explicitly define:

per-worker timeout
one retry max for transient failure
synthesis behavior when some workers fail
whether failed workers contribute error summaries into the final pass

Fifth, do not derive SWARM_MAX_WORKERS too literally from llamaBatchSize. max_num_seqs is an engine concurrency control, but it is not the same thing as a safe user-facing worker ceiling, especially if workers have different prompt sizes and some requests are long-context. I would use a profile-specific swarm cap that is empirically smaller than the raw vLLM limit.

#112 — vLLM Swarm-Mode Profile Tuning

This plan is directionally right. Swarm traffic is a different shape from a normal long-context solo agent, so tuning for many short concurrent requests makes sense.

What’s good
--enable-chunked-prefill is a sensible fit for mixed-length concurrent work, and vLLM exposes chunked-prefill controls plus related knobs like --max-num-batched-tokens, --max-num-partial-prefills, --max-long-partial-prefills, and --long-prefill-token-threshold. requests, so the orchestration layer can actually mark orchestrator/synthesis requests differently from worker traffic. exists but is deprecated in newervllm servedocs. is deprecated and logging is already off by default in some serve modes. -prefill-token-threshold -num-batched-tokens is set above model length in some cases. context size `, and vLLM says non-zero priority will error if the served model is not using priority scheduling. may help throughput, but it also narrows safety margin when many sequences are live. acing “Swarm-ready” on profile cards is smart because users should know which tiers are meant for parallel capacity before they launch
What I’d improve

1. Make the proxy resilient, not just thin.
GET /api/sessions/:id/swarm-status should not fail noisily every time:

the instance is still provisioning
Claw Runner is restarting
the port mapping is briefly stale

FLOATR already has a session lifecycle with provisioning/starting/ready states and periodic syncs.

So I’d make the proxy return:

unavailable
starting
stale
live

instead of just surfacing raw fetch failure.

2. Cache the last known swarm snapshot in the API server.
This is the biggest improvement I’d make.

Do not make the cockpit depend entirely on the live /swarm/status call to the instance. Since #111 already pushes swarm lifecycle events and status snapshots to OMNIQL_CALLBACK_URL, the API server should store:

last known active swarm status
last decomposition
last synthesis result summary

Then the cockpit can show useful state even if the runner is temporarily unreachable.

3. Add a real “historical but quiet” state.
Right now the task says:

tab appears when swarm becomes active
disappears or shows empty state when none has run

I’d separate:

Never swarmed this session
Swarmed earlier, idle now
Swarm active now

That gives users a much clearer picture.

4. Abort should have confirmation and possibly role gating.
Even if it is an emergency-only affordance, one click to kill all workers is risky. I’d add:

confirm dialog
maybe hide behind a danger confirmation phrase
maybe only show to session owner / privileged user in team sessions

FLOATR team sessions already have distinct member access and credentials, so this is worth thinking about.

5. The badge should reflect both progress and failure.
Instead of only 12 / 48 done, consider:

12/48
12 done · 2 failed
Synthesising
Aborted

That will make the cockpit easier to scan.

6. Show the orchestrator’s top-level intent, not just subtask rows.
The first section should include:

why it parallelized
how many workers it chose
maybe the decomposition confidence/reason

Otherwise users may see a tree but not understand the model’s decision.

7. The “Swarm-ready” badge should reflect actual safe swarm capacity, not just Pro/Ultra branding.
If #112 introduces a dedicated swarmWorkerCap, use that in the tooltip:

“Swarm-ready: up to 64 workers”
“Swarm-ready: up to 150 workers”

That is better than a vague capability label.
