This is a strong plan. The shape is right, and it fits FLOATR’s current architecture well: API server orchestration, generated onstart.sh, session lifecycle states, memory service, and cockpit views.

My take:

#38 is solid, but I would tighten 9 things before handing it to Replit.
#39 is good and mostly ready once #38 is corrected.

The biggest issues to fix in #38
1. You’re mixing up trust tier and install risk

Right now the plan uses:

virtual/config/hooked/binary/networked as a “trust tier”
Tier 0 / Tier 1 / Tier 2 / Tier 3 in the UI as trust badges

Those are two different concepts and should not be collapsed.

I’d separate them into:

trustTier: floatr_native | reviewed | user_approved | experimental
installRisk: virtual | config | hooked | binary | networked

That will save you a lot of confusion in ranking, review, and UI.

2. enabled = false is not enough for review state

You need to distinguish:

imported but not reviewed
reviewed and rejected
reviewed and approved but disabled
approved and enabled

So add a real review state, such as:

reviewStatus: pending | approved | rejected
plus reviewedAt, reviewedBy later if needed

Without this, the Imported and Disabled tabs will get muddled immediately.

3. The first-party bundles reference skills that do not exist yet

This is the biggest spec mismatch.

You list bundles like:

repo-graph
blast-radius
checkpoints
role overlays
decision log

But your backend scope explicitly says:

Repo Intelligence is out of scope
Memory 2.0 is out of scope
evals are out of scope

So either:

reduce the launch bundles to only skills you are actually implementing now, or
define those missing ones as explicit placeholder virtual skills with no runtime engine yet

Otherwise Replit will build a bundle system full of fake capabilities.

My recommendation: keep v1 bundles real and smaller.

Example:

floatr-builder: karpathy-doctrine, flow-router, memory-compact, lean-compression
floatr-reviewer: karpathy-doctrine, flow-router, focused-memory, one-line-review
floatr-debugger: debug-flow, checkpoints-lite, focused-memory, compact-response
floatr-team-studio: shared-memory-lite, flow-router, decision-log-lite, lean-compression
4. Your session schema changes are incomplete

You say:

POST /api/sessions calls the bundle compiler … stores the bundleId on the session row

But your schema changes do not add any new columns to sessions.

You need to decide one of these paths:

Option A: keep sessions clean

store everything in session_skills
do not add bundleId to sessions

Option B: add explicit session columns

taskMode
tokenMode
activeBundleId
maybe repoFingerprintJson

I would lean toward Option B, because session detail pages and dashboard filters will want this directly.

5. Repo fingerprinting is underspecified

This is another major gap.

Your bundle compiler expects:

repoLangs
repo/task context
possibly repo kind

But FLOATR’s current launch flow does not inherently know what repo is being worked on. The spec says code-server opens /workspace/projects, but not that a repo is already present there.

So you need to define one of these:

POST /api/sessions accepts repoUrl
or accepts a precomputed repoFingerprint
or compile happens after the repo is cloned
or fallback bundle is used when no repo exists yet

I would add to session launch:

repoUrl?: string
repoBranch?: string
repoFingerprint?: { langs, framework, monorepo, tests }

And make the compiler gracefully degrade to a generic bundle if repo context is missing.

6. Import pipeline should support one source → many skills

A GitHub repo can contain:

multiple skills/*
a CLAUDE.md
commands
hooks
agent files

So POST /api/skills/import { url } should probably return:

one skill_source
many skills

Not one imported repo = one skill.

That also means your normalizer should output an array:

normalizeSource(...) => FloatrSkillManifest[]

This will matter a lot for repos like claude-flow.

7. Don’t pass lots of skill fragments as env vars

This part worries me:

export FLOATR_SKILL_<N>=<base64>

That is fragile. You can hit shell/env size limits or quoting problems fast, especially with 3–7 skills and rich manifests.

Better:

build a single bootstrap JSON blob server-side
compress/base64 once if needed
pass one env var like FLOATR_ACTIVE_BUNDLE_B64
or better yet write one heredoc block directly in the generated wrapper script

Then onstart.sh decodes one file:

/workspace/.floatr/skills/active-bundle.json

From that, it can derive prompt fragments locally.

That is much safer.

8. “Activate bundle on a running session” needs a real runtime contract

POST /api/skill-bundles/:id/activate sounds nice, but what does it actually do on a live instance?

Right now, bundle activation is boot-time injection. So for running sessions you need one of:

Next restart only
Live file rewrite + runner reload
Control endpoint on instance
Claw Runner watches active-bundle.json and hot-reloads

If you do not define this, the endpoint will be fake.

My suggestion:

in Phase 0/1, make it “activate for this session and push updated bundle file”
Claw Runner reloads from disk every N seconds or on signal
code-server users see updated bundle in prompt tooling only after reload

If that is too much, label the endpoint next-launch only for now.

9. litellm proxy is probably the wrong place for tool manifest slimming

FLOATR_TOKEN_MODE absolutely makes sense.
But “read by Claw Runner + litellm proxy to apply tool manifest slimming” feels off.

LiteLLM is mainly a compatibility/proxy layer in FLOATR’s current design. The actual tool surface and prompting logic likely belong in:

Claw Runner
prompt assembly
maybe code-server side config

So I would define token mode effects as:

prompt verbosity
memory retrieval depth
active skill count
response style directive
available tool manifest

And keep those in the agent/runtime layer, not the proxy.

What I would change in the acceptance criteria

I’d rewrite a few “Done looks like” items.

Replace this:

Every skill in the DB has a manifest_json column storing a valid manifest

Because your table design puts manifestJson in skill_versions, not skills.

With this:

Every imported or seeded skill has at least one skill_versions row containing a valid manifestJson, and skills.currentVersionId or equivalent lookup exists

If you do not want currentVersionId, then make sure the API clearly resolves latest version.

Replace this:

Four first-party “Tier 0” launch bundles seeded at server startup

With this:

Four first-party FLOATR-native bundles are seeded at startup, and every included skill is either fully implemented in Phase 0/1 or explicitly marked as a virtual placeholder

Add this:

POST /api/sessions accepts taskMode, tokenMode, and either repoUrl or repoFingerprint; compiler falls back gracefully when repo context is absent

Add this:

Imported sources may yield multiple skills; import response returns source metadata plus created skills array

What I like a lot in #38

These are good calls:

separate Drizzle tables instead of overloading sessions
normalizer service as its own layer
deterministic bundle compiler
OpenAPI first
idempotent seed behavior
conflict detection
token modes as real profiles

That overall structure is right.

Feedback on #39

This one is mostly good. Just a few adjustments:

1. The review gate should use reviewStatus, not infer from enabled

Otherwise Imported vs Disabled will be messy.

2. Use POST /api/skill-bundles/compile, not GET

Your UI steps mention GET /api/skill-bundles/compile?... in one place, but the backend task defines POST. Keep it POST only.

3. “Approve directly for virtual skills” is fine, but still show provenance

Even for text-only skills, the review card should always show:

source repo
pinned commit SHA
license
extracted manifest summary

That keeps trust high.

4. “Replace skill” might be too much for the first dashboard pass

It is useful, but I’d make it secondary to:

view active bundle
approve/reject imports
choose task mode/token mode
override bundle at launch

If time gets squeezed, “replace skill on a live session” is the easiest thing to slip.

5. The feedback endpoint belongs in #38, not #39

The UI depends on it, so backend should own it now.

My recommended cleanup before you send this over

I would make these five edits immediately:

Split trustTier from installRisk
Add reviewStatus
Fix the first-party bundles so they only include real or clearly-placeholder skills
Decide how repo fingerprinting actually enters session launch
Replace many FLOATR_SKILL_<N> env vars with one active bundle bootstrap file
Bottom line

This is a very good plan, but right now it has a few hidden contradictions that will cause rework:

review state vs enabled
trust vs install risk
fake bundle capabilities vs actual scope
repo fingerprinting without repo input
live activation without reload semantics

Fix those now and it becomes a genuinely implementation-ready brief.
