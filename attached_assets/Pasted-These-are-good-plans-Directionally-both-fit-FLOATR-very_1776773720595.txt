Comments on #41 — Context Shield + Working State Continuity

This is the right feature, but it needs firmer boundaries than #40.

What I like
Excellent direction
Strong tool list
Good use of token mode as routing aggressiveness
Working-state journal is a very good idea
Stats/doctor are smart additions
Great default skills to add
What I would tighten
1. “Shield” is not the same as “sandbox”

This is the biggest wording issue.

A subprocess on the user’s rented instance is not a real security sandbox by default. It is a context shield, not a strong isolation boundary.

So I would avoid language that implies hard security guarantees unless you are actually doing real containerization/seccomp/ulimits/FS restrictions.

Rename the mental model to:

shielded execution
bounded-output subprocess
artifact-backed analysis

That is more accurate.

2. Decide whether continuity is within-session or cross-session

Right now you store state at:

/workspace/.floatr/session-state.db

That is fine for:

compaction
reconnect
long-running active session

But it is not enough if the session is destroyed, because FLOATR sessions are ephemeral by default.

So you need to choose:

Option A
This task only guarantees continuity within the same live session.

Option B
Important state snapshots are mirrored back to the API server / memory layer for cross-session restore.

I strongly recommend:

make #41 guarantee in-session continuity
optionally mirror compact summaries to existing memory later

Do not pretend local SQLite alone gives cross-session persistence.

3. Separate working state from long-term memory

You already have a memory system on the API side.

So define the split clearly:

working state = task-local, volatile, operational
memory = durable, cross-session, searchable, summarized

Without that separation, Replit may duplicate memory features into the session DB and create confusion.

4. Auto-routing based on “expected output size” is underspecified

Before execution, you often do not know output size.

So define routing in two stages:

pre-routing by operation type and heuristics
post-routing truncation/artifact persistence if output exceeds thresholds

That is much more realistic.

5. The tool surface is probably too big for first pass

All five tools are good, but for MVP I would prioritize:

floatr_execute
floatr_execute_file
floatr_batch_execute

Then add:

floatr_index
floatr_search

only if they are clearly different from your Repo Intelligence stack. Otherwise you risk overlapping concepts with #40.

6. Define the restore injection point

You say:

a compact restore block is injected before work continues

Good, but where?

Define it precisely:

on Claw Runner task start
on agent session init
after compaction event
on reconnect
before prompt assembly

That will make the implementation much cleaner.

7. Stats/doctor should be internal/runtime-first

I like them, but I would not make HTTP endpoints the first priority.

For MVP:

make them runtime tools / local commands first
add API endpoints only if dashboard needs them immediately

Otherwise you are adding observability surface before proving the core loop.

8. Add artifact retention rules

Shielded execution will create files.

You need:

storage path
retention window
max artifact size
cleanup policy
whether retained artifacts are searchable

Without that, disks will fill up quietly.

9. Namespace actor identity in the event journal

Even if multi-user merge is out of scope, team sessions already exist.

So journal rows should still capture:

actor type (user, agent, system)
actor id / workspace
timestamp
task id if present

That will save pain later.

10. Do not let #41 swallow half of Memory 2.0

The risk here is scope creep.

Keep #41 focused on:

bounded-output execution
routing
working-state snapshots
restore

Do not let it grow into:

contradiction detection
hierarchical recall
promotion/decay
rich memory semantics

Those belong in the later Memory 2.0 phase.
