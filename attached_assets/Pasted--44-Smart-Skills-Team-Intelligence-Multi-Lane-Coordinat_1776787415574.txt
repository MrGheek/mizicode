#44 - Smart Skills: Team Intelligence + Multi-Lane Coordination

What & Why

FLOATR already supports team sessions where one rented GPU instance can host multiple isolated IDEs plus a shared workspace, with per-member credentials and shared access patterns. It also already has Claw Runner, persistent memory, and a cockpit model for session state.

The next gap is coordination.

Right now, Smart Skills, Repo Intelligence, Context Shield, and Memory Governance make the session smarter. This task makes the team working inside that session smarter.

The goal is to support a shared codebase where:

one teammate is doing UX
another is debugging
another is working on API/backend/review
all are sharing one hourly GPU session
all need the same repo intelligence and efficiency guarantees
but each needs different working state, different skill overlays, and lightweight coordination so they do not step on each other

This task adds:

per-member lane overlays
shared vs personal memory scopes
task ownership / file ownership tracking
overlap and blast-radius warnings between lanes
heavy-job contention management for one shared GPU session
a coordination layer that shows who is working on what and where conflicts are likely
Done looks like
Layered lane model exists

Each team session now has four coordination layers:

Session Core — always-on efficiency and contention rules
Shared Repo Layer — common repo graph, architecture summary, conventions
Lane Overlay — per-member task-specific skill bundle
Coordination Layer — ownership, overlap, and handoff state
Per-member lane overlays

Each team member in a shared session can have their own active overlay, for example:

UX / frontend lane
debug lane
API/backend lane
review/refactor lane

Lane overlays affect:

active skills
retrieval weighting
working-state snapshots
token mode defaults
context emphasis

The session core remains common, but the lane overlay differs per teammate.

Shared vs personal memory scopes

Memory retrieval and persistence now support:

session-core
repo-shared
lane-user
task-specific

This means:

shared architectural facts and conventions are reusable by everyone
local hypotheses, active debugging notes, and personal working state stay in the correct lane unless explicitly promoted
Ownership and overlap tracking

FLOATR can track:

who owns which task
who is currently working on which files/symbols/modules
likely overlap between teammates
likely merge risk / blast-radius collision

If two lanes touch overlapping files or symbols, FLOATR raises a coordination warning.

Lane-aware repo retrieval

Repo Intelligence remains shared, but retrieval is lane-aware:

UX lane prioritizes components, routes, styles, previews, design tokens
debug lane prioritizes failing tests, logs, recent changes, callers/callees
backend lane prioritizes services, handlers, DB paths, contracts
review lane prioritizes diffs, blast radius, risky files, tests
Heavy-job contention management

Because the whole team shares one hourly GPU session, expensive jobs are coordinated:

graph rebuilds
embedding/index refreshes
deep log analysis
large batch retrieval
long shielded execution tasks

FLOATR applies:

queueing
priority levels
“heavy job in progress” awareness
per-lane fairness rules

Lightweight operations continue freely; heavyweight operations are coordinated.

Coordination state exists

FLOATR stores a lightweight shared coordination state for the session:

active lanes
current task per lane
status (active, blocked, review-needed, ready-to-merge)
claimed files/modules
last update time
handoff notes
Handoff support exists

A teammate can mark:

“I’m blocked on this”
“Needs review”
“Safe to merge”
“Watch these files”
“Related to your lane”

This becomes part of the shared coordination layer, not just free-text memory.

Smart Skills bundle compiler becomes lane-aware

The bundle compiler can now produce:

one session-core bundle
one shared repo bundle
one overlay bundle per lane

Imported skills can be applied per lane, but they do not override the session core by default.

Out of scope
Full real-time collaborative editing
Git branch orchestration / automatic merges
Fine-grained code locking
Full org-level team analytics across many sessions
Visual repo heatmap UI
Autonomous manager/dispatcher agent
Cross-session staffing recommendations
Role-specific HR/accounting permissions
Multi-session federated lane coordination
Steps
1. Add team lane data model

Add new tables such as:

session_lanes
lane_assignments
lane_state
lane_claims
lane_handoffs
lane_heavy_jobs

Suggested fields:

sessionId
memberName or member identifier
laneType
taskMode
status
overlayBundleId
tokenMode
claimedPathsJson
claimedSymbolsJson
handoffNotes
priority
updatedAt

Keep existing session/team primary key types unchanged.

2. Define lane types and overlay policy

Create a lane policy layer with initial supported lane types:

ux
debug
backend
review
general

Define:

default overlay bundle per lane type
retrieval emphasis per lane type
default token mode by lane type
escalation rules for overlap/conflict
3. Add lane-aware memory scopes

Extend memory handling so each saved item can live in:

session_core
repo_shared
lane_user
task

Define retrieval precedence:

current lane task
current lane state
shared repo memory
session core
user/operator
global
4. Add ownership / claim system

Implement services that allow a lane to claim:

files
modules
symbols
tasks

Claims are soft, not hard locks.
If another lane touches overlapping claims, FLOATR creates a coordination warning instead of blocking work.

5. Add overlap + collision detection

Use repo graph data from #40:

compare claimed files/symbols
compare blast radius between lane changes
detect likely collision risk
raise warnings with confidence score

Examples:

UX lane edits a shared button component while debug lane is tracing failures through the same component tree
backend lane changes an API contract that review lane is currently evaluating
6. Add lane-aware bundle compilation

Extend skills-bundler.ts / skills-ranker.ts so team sessions compile:

a mandatory session core
a shared repo bundle
one overlay bundle per lane

Session core always includes the house efficiency constraints.
Lane overlays add task-specific behavior without replacing the core.

7. Add heavy-job contention scheduler

Build a small scheduler for GPU-expensive coordination jobs:

queue deep retrieval/index jobs
prioritize active debugging / review-critical work
expose queued, running, deferred, completed
prevent one lane’s heavy job from starving the whole session

This scheduler should sit alongside Claw Runner work orchestration, not inside LiteLLM.

8. Add coordination APIs

Add APIs such as:

GET /api/sessions/:id/lanes
POST /api/sessions/:id/lanes
PUT /api/sessions/:id/lanes/:laneId
POST /api/sessions/:id/lanes/:laneId/claim
POST /api/sessions/:id/lanes/:laneId/handoff
GET /api/sessions/:id/coordination
GET /api/sessions/:id/conflicts
POST /api/sessions/:id/heavy-jobs
GET /api/sessions/:id/heavy-jobs
9. Add first-party coordination skills

Add first-party manifests such as:

team-coordination-core
lane-awareness
shared-repo-memory
contention-management-core
handoff-discipline

These should rank high for:

team sessions
shared workspaces
long-running multi-task sessions
large repos with overlapping ownership
10. Update cockpit / launch surfaces later

Backend should support future UI work, but this task does not require the full coordination dashboard.
It only needs the APIs, persistence, compiler logic, and coordination services to exist cleanly.

11. OpenAPI + codegen

Add all request/response schemas to lib/api-spec/openapi.yaml and regenerate Zod/client code.

Design note

Do not model a team session as “one bundle for everyone.”

Model it as:

one session core
one shared repo brain
one lane overlay per teammate
one coordination layer

That matches FLOATR’s existing team-session shape, where one session can already host multiple isolated IDEs and a shared workspace on a single GPU-backed environment.
