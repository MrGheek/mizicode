Now I have the full picture. Here's how a clean separation works.

The Right Approach: Same Codebase, Separate Build Profiles
The monorepo already switches modes via environment variables and a provider field. The cleanest path is to add a local build profile that produces an entirely different distribution artifact — same source, but the cloud provider code (Vast.ai, Fly.io, vLLM) is tree-shaken out at build time via esbuild define flags.

Cloud Edition                    Local Edition
─────────────────                ─────────────────────────
Fly.io deploy                    .deb / tarball / ARM64 image
vastai.ts  ✓                     vastai.ts  ✗ (stripped)
fly.ts     ✓                     fly.ts     ✗ (stripped)
vLLM       ✓                     vLLM       ✗ (stripped)
PostgreSQL ✓                     SQLite     ✓
onstart.sh ✓                     local-start.sh ✓
Cloud API keys required          Zero cloud dependencies

How the Split Works in Practice
1. Build flag gates cloud code out

MIZI_DISTRIBUTION=local pnpm build

esbuild's define replaces MIZI_DISTRIBUTION at compile time — dead code elimination removes all Vast.ai/Fly.io imports from the bundle entirely. Zero cloud SDK bytes in the Pi binary.

2. A new local provider alongside vastai and fly
The provider field in the sessions table already governs which path runs. A local provider skips all provisioning — the Pi is the workspace.

3. SQLite replaces PostgreSQL for local
The lib/db package already uses Drizzle ORM — adding a SQLite driver alongside the existing Postgres driver is a config swap, not a rewrite.

4. Separate release artifact
The local build produces a mizi-local-arm64.tar.gz containing:

Built API server (dist/index.mjs)
Static dashboard assets
systemd unit files
local-start.sh one-command setup (installs Ollama, HailoRT, pulls model)
config.env.template (no cloud keys needed)
5. Separate versioning & release channel
A CHANGELOG.local.md and separate GitHub release tag (v1.0.0-local) keeps the two editions independently versioned while sharing the same git history.
