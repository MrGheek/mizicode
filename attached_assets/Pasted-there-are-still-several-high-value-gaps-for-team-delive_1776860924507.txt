This task adds the next GitHub/CICD layer:

merge queue support so busy branches stay green while multiple PRs land
GitHub Projects automation so issues and PRs automatically flow into a shared team board
PR preview deployments for dashboard/frontend changes
CodeQL code scanning
workflow linting + Actions hardening
artifact attestations for build provenance
secret scanning / push protection rollout guidance

GitHub’s merge queue requires workflows to also trigger on merge_group; GitHub Projects supports built-in automation for auto-adding and updating items; GitHub Environments support deployment history, protection rules, and environment-scoped secrets; CodeQL provides code scanning alerts; GitHub Actions supports artifact attestations for provenance; and GitHub secret scanning push protection can block pushes containing supported secrets before they land.

FLOATR is a monorepo with distinct backend, dashboard, docker, and shared library areas, so these additions should respect the repo structure and team workflow rather than assume a single-package app.

Done looks like
Required CI workflows run on pull_request, push to main, and merge_group, so merge queue can be enabled safely.
A GitHub Project exists for FLOATR engineering, and new issues/PRs are automatically added to it with built-in automation.
Frontend/dashboard PRs get a preview deployment URL posted back to the PR, using a GitHub Environment such as preview.
CodeQL runs on PRs and main-branch pushes and surfaces code scanning alerts in GitHub.
GitHub workflow files are linted automatically with actionlint, and workflow security is hardened with explicit permissions, SHA-pinned actions where practical, and repository/org policy compatibility for requiring full-length commit SHAs.
Release or Docker artifact workflows generate artifact attestations for provenance.
The repo contains documentation and optional checks for enabling secret scanning push protection at the repository level.
Out of scope
Replacing the existing Docker build workflow logic
Changing application code in artifacts/api-server or artifacts/dashboard
A full custom project-management app inside FLOATR
Provider-specific preview infra beyond what is needed to make preview deploys work
Cross-repo org automation beyond this repository
Fully automated release trains or auto-merge
Enterprise-wide security settings rollout outside this repository, except where documented as manual follow-up
Steps
1. Add merge queue support to CI

Update existing CI workflows so required checks run on:

pull_request
push to main
merge_group

Add concurrency with cancel-in-progress for PR workflows where appropriate.

Keep workflow naming stable so these checks can be used in branch protection / rulesets. GitHub requires merge_group triggers for required checks to run under merge queue.

2. Add GitHub Projects automation support

Create repo documentation and optional helper automation for a FLOATR engineering project with fields such as:

Status
Component
Priority
Owner
Release / Milestone

Assume GitHub’s built-in Project workflows will be used for:

auto-adding new issues and PRs
setting status transitions like Todo / In Review / Done
archiving completed work if desired

Add a small .github/projects.md or similar ops doc that explains the intended board configuration and automation filters. GitHub Projects supports auto-add workflows and built-in automations for status updates and archiving.

3. Add PR preview deployment workflow for dashboard changes

Create a workflow such as preview-dashboard.yml that:

triggers on PRs touching artifacts/dashboard/** and any shared files required for dashboard builds
builds the dashboard
deploys a preview to the configured preview target
uses a GitHub Environment named preview
comments the preview URL on the PR
cleans up or marks superseded previews when a newer commit arrives

Keep deployment logic provider-agnostic where possible, but structure the workflow to use Environments, environment secrets, and deployment records cleanly. GitHub Environments support secrets, protection rules, and deployment history.

4. Add CodeQL workflow

Create codeql.yml that runs on:

pull_request
push to main
optional weekly schedule

Start with JavaScript/TypeScript coverage. If repo structure later justifies it, extend to additional languages/scripts.

Use GitHub’s standard CodeQL setup so alerts appear in the Security tab and pull requests. GitHub documents CodeQL/code scanning as a built-in code scanning path.

5. Add workflow linting and Actions hardening

Create a workflow like workflow-hygiene.yml that:

runs actionlint on .github/workflows/**
validates workflow syntax and common mistakes
can optionally check YAML formatting if tooling already exists

Also harden all workflows by:

adding explicit permissions: blocks
pinning third-party actions to full SHAs where practical
keeping reusable workflows compatible with GitHub’s Actions security settings

GitHub supports requiring actions to be pinned to full-length SHAs at the repository/org level, and actionlint is a static checker for workflow files.

6. Add artifact attestations to release/build workflows

Extend the existing Docker/release workflow surface so built artifacts generate provenance attestations.

At minimum:

identify which build outputs should be attested
add the necessary workflow permissions
add attestation generation to the build/release path
document how attestations are verified

GitHub Actions supports artifact attestations for binaries and container images and documents provenance generation and verification.

7. Add secret-scanning / push-protection rollout docs and optional guard workflow

Create a short ops doc such as .github/security-hardening.md that explains:

enabling secret scanning and push protection in repository settings
what contributors should do when a push is blocked
how bypasses are reviewed
which secrets are especially relevant to FLOATR (API keys, tokens, cloud creds, etc.)

Optionally add a lightweight workflow or checklist that verifies the repo’s security documentation is present, but do not try to “implement push protection in code” because it is primarily a repository setting. GitHub push protection blocks pushes containing supported secrets before they reach the repository.

8. Add branch/ruleset follow-up documentation

Add a docs/github-ops.md or similar file documenting the intended GitHub settings after the workflows land:

enable merge queue on main
make CI checks required
require CODEOWNERS review where desired
optionally require actions to be pinned to SHAs
enable secret scanning / push protection
configure the Project automations
configure the preview Environment secrets

This task should ship the repository-side files plus the human-readable runbook for the GitHub settings that must be toggled manually.

Additional implementation guidance
Respect FLOATR’s monorepo layout when choosing path filters and component ownership. Backend, dashboard, docker, and shared lib areas should not all trigger the same workflows unnecessarily.
Prefer additive changes over reworking the existing Docker build pipeline.
Keep preview deployment wiring isolated so backend-only changes do not incur preview deployment cost.
Keep release/build provenance and security workflows lightweight enough that they do not slow down everyday PR iteration unnecessarily.
Manual repo settings after merge

After the code/config is merged, document and then apply these in GitHub:

enable merge queue / branch rules or rulesets on main
configure required checks
create/configure the FLOATR Project and its built-in automations
create the preview environment and add needed secrets
enable CodeQL/code scanning if required by plan/repo type
enable secret scanning and push protection if available for the repo plan
consider enabling “Require actions to be pinned to a full-length commit SHA” once workflows are pinned and stable.
