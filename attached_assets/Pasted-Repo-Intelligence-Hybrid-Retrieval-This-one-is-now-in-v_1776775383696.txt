Repo Intelligence + Hybrid Retrieval

This one is now in very good shape. The async job model, Claw Runner ownership, graceful fallback path, local-first embeddings, explicit ignore rules, and search filters all fix the earlier gaps.

The only things I’d still tighten:

Pick the embedding strategy now, not later.
“Instance-local model” is right, but Replit will move faster if you specify the default implementation up front:
either a small local embedding model running on CPU
or a lightweight local embedding service on the instance
Otherwise they may lose time debating architecture mid-build.
Add reindex + stale-state semantics.
You have /repo/index, but I’d define:
what triggers a full rebuild vs incremental refresh
whether a second index request cancels, dedupes, or queues
how stale indexes are marked in API responses
Return score breakdowns in /repo/search.
Since the ranking is hybrid, expose:
combined score
lexical score
semantic score
graph score
confidence
That will make debugging and tuning much easier.
Put hard runtime limits on indexing jobs.
You already added ignore patterns and file caps, which is great. I’d also add:
max indexing duration
max chunk count
max embedding batch size
fallback to graph-lite / lexical-only mode if exceeded
Define what “repoPath omitted” means in team sessions.
You say owner workspace root is default, which is good. I’d make that explicit in the acceptance criteria too, since team sessions already split workspaces.
