Here’s a detailed product spec for FLOATR Smart Skills that builds directly on FLOATR’s current architecture: rented GPU sessions, local model serving through vLLM + LiteLLM, browser IDEs, Claw Runner, SQLite FTS5 memory, team sessions, and boot-time environment control.

FLOATR Smart Skills — Detailed Product Spec
1. Product summary

FLOATR Smart Skills is a capability layer that automatically selects, installs, and activates the best skill bundle for a coding session based on:

the repo
the task
the model/profile
the user/team mode
the token budget
past session outcomes

The goal is to make FLOATR feel like a self-assembling coding cockpit, not just a GPU-hosted IDE. FLOATR already controls session launch, model selection, persistent memory, and startup orchestration, which makes it the right place to compile and inject skill bundles at session boot.

2. Why this fits FLOATR

FLOATR already has the exact primitives Smart Skills needs:

local inference on the rented instance via vLLM and LiteLLM
a controlled boot pipeline via onstart.sh
persistent cross-session memory using SQLite FTS5
Claw Runner for longer-lived agent workflows
team sessions with isolated workspaces and shared context
an API server that already manages profiles, sessions, templates, scheduler, and memory routes

That means Smart Skills does not need to be a loose plugin directory. It can be a ranked runtime system with provenance, policy, performance measurement, and session-specific assembly.

3. Product goal

Make FLOATR automatically answer this question at launch:

“Given this repo, this task, this model, and this team setup, what is the smallest set of skills that will produce the best result?”

Not “what skills exist?”
Not “what is popular on GitHub?”
But “what should be active right now?”

4. What Smart Skills should borrow
Doctrine layer

The Andrej/Karpathy skill repo is strongest as a behavior contract: think before coding, prefer simplicity, make surgical changes, and define verifiable success criteria. That is ideal as FLOATR’s default doctrine bundle for coding sessions.

Workflow layer

claude-flow is strongest as a routing system: one /flow entry point, a routing table for overlapping tools, built-in plan/TDD/checkpoint/build-fix flows, and continuous discovery/update of new tools. FLOATR should borrow the routing discipline, not just the command name.

Memory layer

CatchMe is strongest as a hierarchical memory model: local/offline SQLite + FTS5, a top-down retrieval path, and one-file agent integration. FLOATR already has persistent memory; the upgrade is to make it tree-structured, summarized, and selectively traversed.

Compression layer

Caveman is strongest as a response compression system with installable hooks/status behavior and published token reduction claims. FLOATR should offer compression profiles rather than copying the persona literally.

Structural code-context layer

code-review-graph is strongest as a repo intelligence system: Tree-sitter parsing, graph storage, blast-radius analysis, and minimal review-set generation. That should become FLOATR’s repo-context engine.

Token governance layer

token-savior is strongest as a budgeted context system: progressive disclosure, contradiction detection, TTL/decay, symbol staleness, ROI scoring, and profile-based tool slimming. FLOATR should adopt these ideas to stop Smart Skills from bloating the prompt and tool surface.

5. Product principles
Normalized, not copied
FLOATR should ingest external skills into a FLOATR manifest format rather than running arbitrary third-party instructions directly.
Bundle-first, not marketplace-first
The key unit is the session bundle, not the individual skill.
Minimal active set
A typical session should activate 3–7 skills, not dozens.
Measured lift
A skill only becomes “best in class” if it improves outcomes in FLOATR.
Private by default
All activation, ranking, memory, and repo analysis should work inside the user’s rented environment or FLOATR control plane.
6. Scope for v1

Smart Skills v1 should support:

importing skill sources from GitHub repos/URLs
parsing SKILL.md, CLAUDE.md, AGENTS.md, command docs, hook docs, and repo metadata
normalizing them into a FLOATR manifest
classifying skills into doctrine/workflow/context/efficiency
generating a recommended session bundle
activating the bundle at session launch
showing active skills and rationale in the cockpit
collecting feedback and outcome metrics
Out of scope for v1
autonomous installation of arbitrary shell hooks from untrusted repos
full weekly GitHub crawling with no review gate
auto-merging upstream changes into active bundles without approval
general-purpose MCP marketplace management

10. Import pipeline
Step 1: fetch and pin

FLOATR imports from a repo URL and records:

repo URL
branch/tag
resolved commit SHA
license
imported files
import timestamp

This matters because the referenced repos do not all use the same license. Several are MIT, while CatchMe is Apache-2.0, so FLOATR needs source provenance and license-aware policy from day one.

Step 2: detect artifacts

Parser looks for:

SKILL.md
skills/*
CLAUDE.md
AGENTS.md
commands/*
hooks/*
install docs
benchmark/eval docs
config files
statusline/hook integrations
Step 3: classify

Normalize each artifact into one or more manifest entries.

Examples:

Karpathy repo → doctrine skill
claude-flow → workflow skill + flow router
CatchMe → memory strategy skill
caveman → compression profile skill
code-review-graph → repo intelligence skill
token-savior → budget/governance skill
Step 4: risk grade

Every skill gets a trust level:

virtual — text instructions only
config — writes local config files
hooked — adds hooks/status integration
binary — installs runtime/program
networked — requires network access
unsafe — arbitrary execution, blocked by default
Step 5: human review

Default policy:

doctrine/workflow/context text skills can be approved quickly
anything that writes hooks or runs binaries requires explicit review

Selection logic

Use a ranking score like:

score = task_fit + repo_fit + model_fit + measured_lift + trust + freshness - token_cost - conflict_risk - install_risk

Examples

For a large TypeScript monorepo review:

repo graph gets boosted
compression gets boosted
full memory gets reduced unless prior context matters

For a small bugfix:

doctrine + simple workflow + lean memory
no heavyweight graph boot unless repo complexity justifies it

For a team session:

shared memory and role-specific overlays get boosted

15. Memory 2.0 for Smart Skills

FLOATR already stores cross-session observations and summaries in SQLite FTS5 and exposes them through dashboard and per-session routes. Smart Skills should upgrade this from a flat searchable log into a tiered, structured memory system.

Memory improvements

Add:

hierarchical rollups: workspace → repo → branch → session → task → symbol
top-down retrieval
contradiction detection
per-type TTL
ROI scoring
symbol staleness invalidation
progressive disclosure lookup

These patterns are strongly supported by CatchMe’s hierarchical retrieval model and token-savior’s TTL, contradiction, ROI, and layered retrieval ideas.

Retrieval contract

Layer 1: summary index
Layer 2: compact search hits
Layer 3: full observation/materialization

That keeps context cheap and predictable.

16. Token modes

Borrow the idea of profile-based capability slimming and expose it as a FLOATR feature.

Modes
Full — all active skills, richer memory, verbose reasoning
Core — default for daily development
Lean — limited tools, compact memory, shorter responses
Ultra — hot-path tools only, progressive discovery

This follows the spirit of token-savior’s profiles and caveman’s compression gains, but FLOATR should present it as a first-class session control.

24. What to build immediately

The best immediate move is:

Build the manifest + importer + manual review + bundle compiler first.

Why:

it turns ecosystem repos into FLOATR-native assets
it avoids unsafe direct execution
it gives you something demoable fast
it creates the foundation for ranking, evals, and future automation

Without that layer, Smart Skills is just “support for random prompt files.”
With it, Smart Skills becomes a real product system.

25. Product positioning

The best positioning line is:

FLOATR automatically assembles the best coding brain for each repo, task, and model — privately, on your own GPU session.

And the moat is:

not the imported repos themselves, but FLOATR’s ability to normalize, rank, bundle, measure, and improve them over time.
