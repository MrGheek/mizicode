The most important issues I found

1. Scheduler-launched sessions appear materially weaker than user-launched sessions.
This is the biggest implementation risk I found.

Manual session creation passes memory proxy settings, callback base URL, session ID, team info, and active skills bundle into buildOnStartScript() . But launchScheduledSession() only passes model params and swarm cap; it does not pass callback, memory proxy, auth token, session ID, team payload, or skills bundle payload .

That likely means scheduler-launched sessions can miss:

live boot callbacks
memory proxy integration
smart skills activation
parity with normal launched sessions

This is the clearest “spec says one thing, runtime behavior may be weaker” gap.

2. Docker workflow behavior does not match the spec.
Your spec says the Docker workflow validates builds on PRs touching docker/**, runs on push to main, and does not auto-push to Docker Hub. The actual workflow is named “Build & Push Docker Image,” triggers on workflow_dispatch and push to main with docker/** path filtering, logs into Docker Hub, and pushes multiple tags automatically .

That is not a minor wording difference. It changes release risk materially.

3. Swarm callback drift still looks risky.
The Claw Runner still derives its swarm callback by replacing /status with /swarm-status on OMNIQL_CALLBACK_URL . Your spec explicitly calls out a known discrepancy around /swarm-status versus /swarm-push . I did not fully verify from the truncated sessions route whether you already added a server alias, so I won’t overclaim. But I would still treat this as a live correctness check item.

4. The spec is already stale in a few important stack declarations.
The spec says React 18 and Zod v4. The repo is on React 19.1.0 and Zod 3.25.x in the workspace catalog . That is not a problem in itself, but it means the spec is no longer a perfect source of truth on the frontend/tooling layer.

5. Naming drift is still present.
There is still a lot of OmniQL language in comments, env vars, auth realms, callback naming, and boot script logs, even though the repo uses FLOATR directories like /workspace/.floatr . This is not just cosmetic. Naming drift causes operator mistakes, confusing logs, and API/env sprawl over time.

Good signs that increase confidence

The API server route mounting is coherent and not spaghetti-like .
Profile seeding is detailed and model-aware, with concrete swarm caps and vLLM flags that match the intended product strategy .
The Dockerfile and onstart logic show real operational understanding rather than generic AI scaffolding .
The dashboard code is not placeholder UI; it is wired to actual API hooks and state models .

My priority ranking

P0

Make scheduler-launched sessions pass the same callback, memory, and skills context as normal launches.
Verify and normalize swarm callback routing end to end.

P1

Decide whether the Docker workflow should be “validate only” or “build and push,” then align repo and spec.
Update the spec for React/Zod/current workflow behavior.

P2

Finish the OmniQL → FLOATR naming cleanup across scripts, env vars, comments, and auth labels.
Review whether there is duplicate route vocabulary like refresh vs sync in the sessions API, because the code definitely has /refresh in the route file and the spec uses /sync .

Bottom line

FLOATR is strong. The repo backs up the ambition in the spec. The architecture is credible, the implementation is non-trivial, and the engineering taste is generally good. The biggest issue is not that the repo is weak. It is that a few operational paths — especially scheduler parity and callback consistency — could undermine trust in a product that is otherwise well built.
